# Triage incidents and enforce SLAs with Gemini, Groq, Google Sheets and Slack

# 1. Workflow Overview

This workflow implements an AI-assisted incident triage and SLA enforcement pipeline in n8n. It receives incident reports via webhook, enriches them with runbook data from Google Sheets, uses two AI agents to classify severity and generate a response plan, logs the outcome for operations and auditability, routes notifications to Slack by priority, and enforces acknowledgment SLAs with escalation paths. It also exposes separate webhook entry points for human acknowledgment and post-incident feedback.

Typical use cases include:

- Centralized intake of operational incidents from internal tools or custom forms
- AI-supported severity classification and early response planning
- Slack-based alerting and war-room creation for critical incidents
- SLA monitoring based on severity
- Audit logging of AI decisions and human feedback for later review

## 1.1 Incident Intake and Normalization

This block receives incident reports over HTTP, immediately acknowledges receipt to the caller, standardizes the payload, and retrieves service-specific runbook information from Google Sheets.

## 1.2 AI Severity Analysis

This block uses a first AI agent to assess incident severity, confidence, urgency, and risk indicators, with Gemini as the primary model and Groq as fallback.

## 1.3 AI Response Planning

This block uses a second AI agent to generate an operational response plan, including actions, escalation tier, notification channels, SLA target, war-room need, and recommended assignee.

## 1.4 Logging and Severity Routing

This block writes the processed incident to the operational tracker and logs AI-decision metadata to an audit sheet, then routes the incident into P1, P2, or P3 handling paths.

## 1.5 P1 Critical Incident Handling

This block sends a critical Slack alert, optionally creates a dedicated Slack war-room channel, waits for the SLA interval, checks acknowledgment status in Google Sheets, and escalates to management if still unacknowledged.

## 1.6 P2 High-Priority Handling

This block posts a high-priority Slack alert, waits one hour, checks acknowledgment, and escalates to engineering leads if nobody has acknowledged the incident.

## 1.7 P3 Informational Handling

This block posts a lightweight notification to Slack and does not trigger automated escalation.

## 1.8 Human Acknowledgment Endpoint

This block provides a webhook for engineers or tools to acknowledge an incident and updates the incident tracker accordingly.

## 1.9 Human Feedback Endpoint

This block provides a webhook for humans to submit feedback on AI severity decisions and updates the audit log.

# 2. Block-by-Block Analysis

## 2.1 Incident Intake and Normalization

### Overview
This block is the main incident entry point. It accepts an HTTP POST request, returns an immediate acceptance response, normalizes the incoming data, and enriches the incident with service-level runbook context from Google Sheets.

### Nodes Involved
- Incident Report Trigger
- Accept and Continue
- Normalize Incident Data
- Fetch Runbook Context
- Build Runbook Context

### Node Details

#### Incident Report Trigger
- **Type and role:** `n8n-nodes-base.webhook`; primary HTTP entry point for new incidents.
- **Configuration choices:**  
  - Path: `incident-report`
  - Method: `POST`
  - Response mode: handled by a separate Respond to Webhook node
- **Key expressions/variables used:** None directly in configuration.
- **Input and output connections:**  
  - No input node; workflow entry point  
  - Outputs to **Accept and Continue**
- **Version-specific requirements:** Type version `1.1`.
- **Edge cases/failures:**  
  - Invalid or empty POST body may still enter the workflow because defaults are applied later
  - If caller expects synchronous processing, this design will not provide final results immediately
- **Sub-workflow reference:** None

#### Accept and Continue
- **Type and role:** `n8n-nodes-base.respondToWebhook`; returns an immediate JSON acceptance response.
- **Configuration choices:**  
  - HTTP response code: `200`
  - Respond with JSON
  - Response body includes `status`, `message`, and `incidentId`
- **Key expressions/variables used:**  
  - `{{ $json.incidentId }}`
- **Input and output connections:**  
  - Input from **Incident Report Trigger**
  - Output to **Normalize Incident Data**
- **Version-specific requirements:** Type version `1.1`.
- **Edge cases/failures:**  
  - At this point, `incidentId` is not yet generated by the normalization node. So the response may return an empty or undefined `incidentId` unless the inbound payload already contains one.
- **Sub-workflow reference:** None

#### Normalize Incident Data
- **Type and role:** `n8n-nodes-base.set`; standardizes the incoming incident fields and generates internal metadata.
- **Configuration choices:**  
  - Generates a synthetic incident ID using timestamp plus random suffix
  - Maps incoming body fields with defaults:
    - `title` default: `Untitled Incident`
    - `description` default: `No description provided`
    - `severity` mapped to `reportedSeverity`, default `P3`
    - `reporter` default: empty string
    - `service` default: `Unknown Service`
  - Adds:
    - `reportedAt` = current timestamp
    - `status` = `Open`
- **Key expressions/variables used:**  
  - `$('Incident Report Trigger').item.json.body.title`
  - `$('Incident Report Trigger').item.json.body.description`
  - `$('Incident Report Trigger').item.json.body.severity`
  - `$('Incident Report Trigger').item.json.body.reporter`
  - `$('Incident Report Trigger').item.json.body.service`
  - `{{ 'INC-' + $now.format('yyyyMMdd-HHmmss') + '-' + Math.random().toString(36).substring(2, 6) }}`
  - `{{ $now.toISO() }}`
- **Input and output connections:**  
  - Input from **Accept and Continue**
  - Output to **Fetch Runbook Context**
- **Version-specific requirements:** Type version `3.4`.
- **Edge cases/failures:**  
  - Random suffix collisions are unlikely but not impossible
  - If the webhook body is not shaped as expected, defaults hide upstream data quality problems
  - Reported severity is accepted without validation at this stage
- **Sub-workflow reference:** None

#### Fetch Runbook Context
- **Type and role:** `n8n-nodes-base.googleSheets`; looks up runbook details by service name.
- **Configuration choices:**  
  - Google Sheets credential: `Google Sheets account 2`
  - Sheet: `Runbooks`
  - Filter: `Service == {{$json.service}}`
  - `onError: continueRegularOutput`, allowing workflow continuation if lookup fails
- **Key expressions/variables used:**  
  - `{{ $json.service }}`
- **Input and output connections:**  
  - Input from **Normalize Incident Data**
  - Output to **Build Runbook Context**
- **Version-specific requirements:** Type version `4.5`.
- **Edge cases/failures:**  
  - Missing `documentId` must be configured manually or the node will fail
  - OAuth permissions or sheet-sharing issues can break lookup
  - Service names must match sheet values exactly enough for the lookup behavior used
  - Because errors continue, downstream logic may proceed with no runbook data
- **Sub-workflow reference:** None

#### Build Runbook Context
- **Type and role:** `n8n-nodes-base.code`; transforms raw runbook row data into a readable context string.
- **Configuration choices:**  
  - Reads the normalized incident from **Normalize Incident Data**
  - Reads first runbook row from current input
  - Builds concatenated context from:
    - `Known Issues`
    - `First Response Steps`
    - `Escalation Contact`
    - `SLA Minutes`
  - Falls back to `No runbook found for this service.`
- **Key expressions/variables used:**  
  - `$('Normalize Incident Data').first().json`
  - `$input.first().json`
- **Input and output connections:**  
  - Input from **Fetch Runbook Context**
  - Output to **Agent 1 - Incident Analyzer**
- **Version-specific requirements:** Type version `2`.
- **Edge cases/failures:**  
  - If Google Sheets returns no row, `runbookRaw` may be empty and fallback text is used
  - If column names differ from expected labels, useful context will be lost
- **Sub-workflow reference:** None

## 2.2 AI Severity Analysis

### Overview
This block uses a prompt-driven AI agent to classify incident severity and produce structured risk analysis. Gemini is configured as the primary model, while Groq is connected as fallback.

### Nodes Involved
- Agent 1 - Incident Analyzer
- Analyzer LLM (Gemini)
- Analyzer Fallback (Groq)
- Parse Analyzer Output

### Node Details

#### Agent 1 - Incident Analyzer
- **Type and role:** `@n8n/n8n-nodes-langchain.agent`; LLM-powered severity analysis agent.
- **Configuration choices:**  
  - Prompt includes title, description, reported severity, service, and runbook context
  - System prompt requires strict JSON output with:
    - `severity`
    - `confidence`
    - `reasoning`
    - `risk_indicators`
    - `matches_known_issue`
    - `urgency_score`
  - Conservative severity rule: if unsure between P1 and P2, choose P2
  - Fallback enabled
  - Output parser expected
- **Key expressions/variables used:**  
  - `{{ $json.title }}`
  - `{{ $json.description }}`
  - `{{ $json.reportedSeverity }}`
  - `{{ $json.service }}`
  - `{{ $json.runbookContext }}`
- **Input and output connections:**  
  - Main input from **Build Runbook Context**
  - AI model inputs from **Analyzer LLM (Gemini)** and **Analyzer Fallback (Groq)**
  - Main output to **Parse Analyzer Output**
- **Version-specific requirements:** Type version `3.1`; requires compatible LangChain nodes in the n8n version used.
- **Edge cases/failures:**  
  - LLM may still return malformed JSON despite prompt rules
  - Provider auth, rate limits, safety filters, and timeouts can fail execution
  - Fallback only helps if configured credentials and model access are valid
- **Sub-workflow reference:** None

#### Analyzer LLM (Gemini)
- **Type and role:** `@n8n/n8n-nodes-langchain.lmChatGoogleGemini`; primary language model provider.
- **Configuration choices:**  
  - Uses default options
  - Connected as AI language model to the analyzer agent
- **Key expressions/variables used:** None in shown config.
- **Input and output connections:**  
  - AI language model output to **Agent 1 - Incident Analyzer**
- **Version-specific requirements:** Type version `1`; requires Gemini credentials to be configured in n8n.
- **Edge cases/failures:**  
  - Missing credentials
  - Quota or rate-limit errors
  - Provider response formatting drift, partially mitigated downstream
- **Sub-workflow reference:** None

#### Analyzer Fallback (Groq)
- **Type and role:** `@n8n/n8n-nodes-langchain.lmChatGroq`; backup model provider for the analyzer.
- **Configuration choices:**  
  - Uses default options
  - Connected as secondary AI language model
- **Key expressions/variables used:** None in shown config.
- **Input and output connections:**  
  - AI language model output to **Agent 1 - Incident Analyzer**
- **Version-specific requirements:** Type version `1`; requires Groq credentials.
- **Edge cases/failures:**  
  - Same class of issues as Gemini: missing auth, unsupported model, rate limits
- **Sub-workflow reference:** None

#### Parse Analyzer Output
- **Type and role:** `n8n-nodes-base.code`; sanitizes and validates AI output, and provides a deterministic fallback if parsing fails.
- **Configuration choices:**  
  - Reads `output` or `text`
  - Strips markdown code fences
  - Extracts first JSON object if mixed text is returned
  - Validates required fields and allowed severity values
  - On success merges parsed analysis with incident context
  - On failure falls back to the reported severity with low confidence
- **Key expressions/variables used:**  
  - `$input.first().json.output`
  - `$input.first().json.text`
  - `$('Build Runbook Context').first().json`
- **Input and output connections:**  
  - Input from **Agent 1 - Incident Analyzer**
  - Output to **Agent 2 - Response Coordinator**
- **Version-specific requirements:** Type version `2`.
- **Edge cases/failures:**  
  - If analyzer output is empty or not parseable, fallback is used
  - `analyzerModel: 'gemini'` is hard-coded on success, even if Groq actually served as fallback
  - Validation only checks a subset of fields, so some malformed but parseable outputs may still pass
- **Sub-workflow reference:** None

## 2.3 AI Response Planning

### Overview
This block uses the analyzed incident to generate a tactical response plan, then converts it into a flattened operational schema for logging and routing.

### Nodes Involved
- Agent 2 - Response Coordinator
- Coordinator LLM (Gemini)
- Coordinator Fallback (Groq)
- Process Response Plan

### Node Details

#### Agent 2 - Response Coordinator
- **Type and role:** `@n8n/n8n-nodes-langchain.agent`; response-planning AI agent.
- **Configuration choices:**  
  - Prompt includes incident ID, service, analyzer output, and runbook context
  - System prompt requires JSON with:
    - `immediate_actions`
    - `escalation_tier`
    - `notification_channels`
    - `sla_minutes`
    - `root_cause_hypothesis`
    - `requires_war_room`
    - `recommended_assignee`
  - Explicit rule: P1 must always require a war room
  - Fallback enabled
- **Key expressions/variables used:**  
  - `{{ $json.incidentId }}`
  - `{{ $json.service }}`
  - `{{ JSON.stringify($json.analysis, null, 2) }}`
  - `{{ $json.runbookContext }}`
- **Input and output connections:**  
  - Main input from **Parse Analyzer Output**
  - AI model inputs from **Coordinator LLM (Gemini)** and **Coordinator Fallback (Groq)**
  - Main output to **Process Response Plan**
- **Version-specific requirements:** Type version `3.1`.
- **Edge cases/failures:**  
  - LLM may hallucinate channels, assignees, or SLA values
  - Prompt says output only JSON, but downstream code still needs to guard against markdown/mixed text
- **Sub-workflow reference:** None

#### Coordinator LLM (Gemini)
- **Type and role:** `@n8n/n8n-nodes-langchain.lmChatGoogleGemini`; primary model for response planning.
- **Configuration choices:** Default options only.
- **Key expressions/variables used:** None.
- **Input and output connections:**  
  - AI language model output to **Agent 2 - Response Coordinator**
- **Version-specific requirements:** Type version `1`; Gemini credentials required.
- **Edge cases/failures:** Provider auth/quota/timeout/safety issues.
- **Sub-workflow reference:** None

#### Coordinator Fallback (Groq)
- **Type and role:** `@n8n/n8n-nodes-langchain.lmChatGroq`; backup model for response planning.
- **Configuration choices:** Default options only.
- **Key expressions/variables used:** None.
- **Input and output connections:**  
  - AI language model output to **Agent 2 - Response Coordinator**
- **Version-specific requirements:** Type version `1`; Groq credentials required.
- **Edge cases/failures:** Same general failure classes as the primary model.
- **Sub-workflow reference:** None

#### Process Response Plan
- **Type and role:** `n8n-nodes-base.code`; parses, validates, and flattens the response plan into a row-friendly operational structure.
- **Configuration choices:**  
  - Strips code fences and extracts JSON if needed
  - Validates plan fields
  - Merges incident metadata, analysis output, and response plan
  - Creates audit fields:
    - `severityOverridden`
    - `analyzerSuccess`
    - `coordinatorSuccess`
  - On parse failure, generates a deterministic default plan based on severity
- **Key expressions/variables used:**  
  - `$input.first().json.output`
  - `$input.first().json.text`
  - `$('Parse Analyzer Output').first().json`
- **Input and output connections:**  
  - Input from **Agent 2 - Response Coordinator**
  - Outputs in parallel to **Log to Incident Tracker** and **Log AI Decision Audit**
- **Version-specific requirements:** Type version `2`.
- **Edge cases/failures:**  
  - If plan JSON is malformed, fallback plan is used
  - Arrays are flattened to semicolon/comma separated strings, which may be harder to parse later by machines
  - Planner can choose channels different from the actual Slack routing logic; the workflow does not dynamically use `notification_channels`
- **Sub-workflow reference:** None

## 2.4 Logging and Severity Routing

### Overview
This block persists the incident and AI decision trail to Google Sheets, then routes the incident into a severity-specific branch.

### Nodes Involved
- Log to Incident Tracker
- Log AI Decision Audit
- Route by AI Severity

### Node Details

#### Log to Incident Tracker
- **Type and role:** `n8n-nodes-base.googleSheets`; appends operational incident records.
- **Configuration choices:**  
  - Operation: `append`
  - Sheet: `Incidents`
  - Writes many columns, including incident data, AI analysis, SLA, and recommendation fields
  - Initializes `Escalated` to `No` and acknowledgment-related fields empty
  - `onError: continueRegularOutput`
- **Key expressions/variables used:** Numerous mappings such as:
  - `{{ $json.incidentId }}`
  - `{{ $json.title }}`
  - `{{ $json.severity }}`
  - `{{ $json.sla_minutes }}`
- **Input and output connections:**  
  - Input from **Process Response Plan**
  - Output to **Route by AI Severity**
- **Version-specific requirements:** Type version `4.5`.
- **Edge cases/failures:**  
  - Missing `documentId` must be set
  - Column names in the sheet must match the configured labels exactly
  - Because errors continue, routing may still proceed even if no incident row is logged, which later breaks acknowledgment checks
- **Sub-workflow reference:** None

#### Log AI Decision Audit
- **Type and role:** `n8n-nodes-base.googleSheets`; appends AI-decision metadata for audit and feedback.
- **Configuration choices:**  
  - Operation: `append`
  - Sheet: `AI_Audit_Log`
  - Stores timestamp, confidence, reported vs AI severity, override status, analyzer/coordinator success, plus empty feedback fields
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ $now.toISO() }}`
  - `{{ $json.severityOverridden }}`
  - `{{ $json.confidence }}`
  - `{{ $json.coordinatorSuccess }}`
- **Input and output connections:**  
  - Input from **Process Response Plan**
  - No downstream connection
- **Version-specific requirements:** Type version `4.5`.
- **Edge cases/failures:**  
  - Same Google Sheets configuration risks as above
  - If feedback updates rely on incident ID matching, the update operation must be correctly keyed in the actual n8n node configuration outside what is visible here
- **Sub-workflow reference:** None

#### Route by AI Severity
- **Type and role:** `n8n-nodes-base.switch`; routes records by AI severity value.
- **Configuration choices:**  
  - Three explicit cases:
    - `P1`
    - `P2`
    - `P3`
  - Fallback output: none
- **Key expressions/variables used:**  
  - `{{ $json.severity }}`
- **Input and output connections:**  
  - Input from **Log to Incident Tracker**
  - Output 1 to **P1 - Send Critical Alert**
  - Output 2 to **P2 - Send High Alert**
  - Output 3 to **P3 - Send Notification**
- **Version-specific requirements:** Type version `3.2`.
- **Edge cases/failures:**  
  - Any unexpected severity value drops out silently because fallback output is disabled
  - Safe only because upstream validation/fallback forces P1/P2/P3
- **Sub-workflow reference:** None

## 2.5 P1 Critical Incident Handling

### Overview
This block handles critical incidents with immediate alerting, optional war-room creation, SLA waiting, acknowledgment checking, and escalation to management when no acknowledgment is recorded in time.

### Nodes Involved
- P1 - Send Critical Alert
- Needs War Room?
- P1 - Create War Room
- P1 - Wait SLA Time
- P1 - Check Acknowledgment
- P1 - Still unacknowledged?
- P1 - Escalate to Manager
- P1 - Flag as Escalated

### Node Details

#### P1 - Send Critical Alert
- **Type and role:** `n8n-nodes-base.slack`; posts a detailed critical incident alert.
- **Configuration choices:**  
  - Sends message to Slack channel `incidents-critical`
  - Includes incident ID, service, title, confidence, reasoning, risk indicators, actions, hypothesis, assignee, and SLA
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ $json.incidentId }}`
  - `{{ Math.round($json.confidence * 100) }}`
  - `{{ $json.immediate_actions }}`
- **Input and output connections:**  
  - Input from **Route by AI Severity**
  - Output to **Needs War Room?**
- **Version-specific requirements:** Type version `2.2`; Slack credentials required even if not explicitly shown in JSON.
- **Edge cases/failures:**  
  - Channel name must exist and app must be allowed to post there
  - Markdown formatting support in Slack may differ from plain text expectations
- **Sub-workflow reference:** None

#### Needs War Room?
- **Type and role:** `n8n-nodes-base.if`; checks whether a dedicated Slack war-room should be created.
- **Configuration choices:**  
  - Condition: `requires_war_room == true`
- **Key expressions/variables used:**  
  - `{{ $json.requires_war_room }}`
- **Input and output connections:**  
  - Input from **P1 - Send Critical Alert**
  - True output to **P1 - Create War Room**
  - False branch has no connected node
- **Version-specific requirements:** Type version `2`.
- **Edge cases/failures:**  
  - False branch ends processing; no SLA wait occurs for P1 incidents where war room is false
  - This is logically inconsistent with the design intent, since all P1s should still be tracked for SLA escalation
- **Sub-workflow reference:** None

#### P1 - Create War Room
- **Type and role:** `n8n-nodes-base.slack`; creates a dedicated Slack channel.
- **Configuration choices:**  
  - Resource: `channel`
  - Channel ID/name expression: `incident-{{$json.incidentId.toLowerCase()}}`
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ 'incident-' + $json.incidentId.toLowerCase() }}`
- **Input and output connections:**  
  - Input from **Needs War Room?**
  - Output to **P1 - Wait SLA Time**
- **Version-specific requirements:** Type version `2.2`; app must have channel-management scopes.
- **Edge cases/failures:**  
  - Slack channel naming restrictions may reject uppercase, special characters, or excessive length
  - Incident IDs include uppercase and hyphens; lowercasing helps, but overall name length should still be checked
  - If channel creation fails, node continues and SLA wait still proceeds
- **Sub-workflow reference:** None

#### P1 - Wait SLA Time
- **Type and role:** `n8n-nodes-base.wait`; delays execution until SLA expiration.
- **Configuration choices:**  
  - Wait duration in minutes = `{{$json.sla_minutes}}`
- **Key expressions/variables used:**  
  - `{{ $json.sla_minutes }}`
- **Input and output connections:**  
  - Input from **P1 - Create War Room**
  - Output to **P1 - Check Acknowledgment**
- **Version-specific requirements:** Type version `1.1`; wait behavior depends on n8n execution mode/storage configuration.
- **Edge cases/failures:**  
  - If `sla_minutes` is null or invalid, wait may fail
  - As noted above, path is unreachable when `requires_war_room` is false
- **Sub-workflow reference:** None

#### P1 - Check Acknowledgment
- **Type and role:** `n8n-nodes-base.googleSheets`; retrieves incident row to check acknowledgment fields.
- **Configuration choices:**  
  - Sheet: `Incidents`
  - Filter: `Incident ID == {{$json.incidentId}}`
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ $json.incidentId }}`
- **Input and output connections:**  
  - Input from **P1 - Wait SLA Time**
  - Output to **P1 - Still unacknowledged?**
- **Version-specific requirements:** Type version `4.5`.
- **Edge cases/failures:**  
  - If the incident was never logged, lookup returns nothing
  - If multiple rows match same Incident ID, result ambiguity can affect downstream logic
- **Sub-workflow reference:** None

#### P1 - Still unacknowledged?
- **Type and role:** `n8n-nodes-base.if`; checks whether `Acknowledged By` is empty.
- **Configuration choices:**  
  - Condition: string empty on `Acknowledged By`
- **Key expressions/variables used:**  
  - `{{ $json['Acknowledged By'] }}`
- **Input and output connections:**  
  - Input from **P1 - Check Acknowledgment**
  - True output to both **P1 - Escalate to Manager** and **P1 - Flag as Escalated**
- **Version-specific requirements:** Type version `2`.
- **Edge cases/failures:**  
  - If lookup returns no row, field may also appear empty and trigger escalation
  - False branch has no downstream node, which is fine for “acknowledged” completion
- **Sub-workflow reference:** None

#### P1 - Escalate to Manager
- **Type and role:** `n8n-nodes-base.slack`; sends management escalation message.
- **Configuration choices:**  
  - Slack OAuth2 credential explicitly set: `Slack account 2`
  - Channel: `management-escalation`
  - Message includes SLA-breach wording and incident metadata from sheet columns
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ $json['Incident ID'] }}`
  - `{{ $json.Service }}`
  - `{{ $json['SLA Minutes'] }}`
- **Input and output connections:**  
  - Input from **P1 - Still unacknowledged?**
  - No downstream node
- **Version-specific requirements:** Type version `2.2`.
- **Edge cases/failures:**  
  - Requires Slack scopes and valid channel access
  - Uses sheet-style column names, so it depends on a successful Google Sheets retrieval
- **Sub-workflow reference:** None

#### P1 - Flag as Escalated
- **Type and role:** `n8n-nodes-base.googleSheets`; marks the incident row as escalated.
- **Configuration choices:**  
  - Operation: `update`
  - Sheet: `Incidents`
  - Sets:
    - `Escalated = Yes`
    - `Escalated At = {{$now.toISO()}}`
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ $now.toISO() }}`
- **Input and output connections:**  
  - Input from **P1 - Still unacknowledged?**
  - No downstream node
- **Version-specific requirements:** Type version `4.5`.
- **Edge cases/failures:**  
  - Update requires proper row matching configuration; if not set correctly, wrong row or no row may be updated
- **Sub-workflow reference:** None

## 2.6 P2 High-Priority Handling

### Overview
This block handles P2 incidents with standard Slack alerting, a one-hour wait, acknowledgment verification, and escalation to engineering leads when the SLA is missed.

### Nodes Involved
- P2 - Send High Alert
- P2 - Wait 1hr
- P2 - Check Acknowledgment
- P2 - Still unacknowledged?
- P2 - Escalate to Manager

### Node Details

#### P2 - Send High Alert
- **Type and role:** `n8n-nodes-base.slack`; posts a high-priority incident message to Slack.
- **Configuration choices:**  
  - Channel: `incidents`
  - Includes incident details, confidence, reasoning, actions, assignee, and SLA formatting in hours/minutes
  - `onError: continueRegularOutput`
- **Key expressions/variables used:**  
  - `{{ Math.round($json.confidence * 100) }}`
  - `{{ Math.floor($json.sla_minutes / 60) }}`
  - `{{ $json.sla_minutes % 60 }}`
- **Input and output connections:**  
  - Input from **Route by AI Severity**
  - Output to **P2 - Wait 1hr**
- **Version-specific requirements:** Type version `2.2`.
- **Edge cases/failures:**  
  - Channel access and Slack auth issues
- **Sub-workflow reference:** None

#### P2 - Wait 1hr
- **Type and role:** `n8n-nodes-base.wait`; delays the check for one hour.
- **Configuration choices:**  
  - Fixed wait: 60 minutes
- **Key expressions/variables used:** None.
- **Input and output connections:**  
  - Input from **P2 - Send High Alert**

# 4. Reproducing the Workflow from Scratch

1. **Create a new workflow** in n8n and name it something like:  
   **Two-Agent Incident Triage with Self-Learning SLA Enforcement**.

2. **Create the incident intake webhook**
   - Add a **Webhook** node named **Incident Report Trigger**
   - Set:
     - Method: `POST`
     - Path: `incident-report`
     - Response mode: `Using Respond to Webhook node`

3. **Add an immediate webhook response**
   - Add **Respond to Webhook** named **Accept and Continue**
   - Connect **Incident Report Trigger → Accept and Continue**
   - Set:
     - Respond with: `JSON`
     - Response code: `200`
     - Body:
       - status = `accepted`
       - message = `Incident report received. Processing started.`
       - incidentId = expression from payload or leave as-is if you later fix the design
   - Recommended improvement when rebuilding: generate the incident ID before responding, otherwise the reply may not contain the final ID.

4. **Normalize incoming incident data**
   - Add a **Set** node named **Normalize Incident Data**
   - Connect **Accept and Continue → Normalize Incident Data**
   - Create these fields:
     - `incidentId` = generated string with timestamp and random suffix
     - `title` = incoming `body.title` or `Untitled Incident`
     - `description` = incoming `body.description` or `No description provided`
     - `reportedSeverity` = incoming `body.severity` or `P3`
     - `reporter` = incoming `body.reporter` or empty string
     - `service` = incoming `body.service` or `Unknown Service`
     - `reportedAt` = current ISO timestamp
     - `status` = `Open`

5. **Prepare Google Sheets data source**
   - Create one Google spreadsheet with these tabs:
     - `Runbooks`
     - `Incidents`
     - `AI_Audit_Log`
   - In `Runbooks`, include at least:
     - `Service`
     - `Known Issues`
     - `First Response Steps`
     - `Escalation Contact`
     - `SLA Minutes`
   - In `Incidents`, create columns matching the append/update mappings used later
   - In `AI_Audit_Log`, create columns matching the audit append/update mappings

6. **Add Google Sheets credentials**
   - Configure a **Google Sheets OAuth2** credential in n8n
   - Share the target spreadsheet with the connected Google account if necessary

7. **Fetch runbook data**
   - Add a **Google Sheets** node named **Fetch Runbook Context**
   - Connect **Normalize Incident Data → Fetch Runbook Context**
   - Configure:
     - Credential: your Google Sheets OAuth2 credential
     - Document ID: your spreadsheet ID
     - Sheet: `Runbooks`
     - Operation: lookup/read row(s) by filter
     - Filter: `Service` equals `{{$json.service}}`
     - Error handling: `Continue on Fail`

8. **Build a runbook context string**
   - Add a **Code** node named **Build Runbook Context**
   - Connect **Fetch Runbook Context → Build Runbook Context**
   - Use code that:
     - Loads normalized incident data
     - Reads the first runbook row
     - Concatenates known issues, first response steps, escalation contact, and SLA minutes into one string
     - Falls back to `No runbook found for this service.`

9. **Add the first AI agent**
   - Add an **AI Agent** node named **Agent 1 - Incident Analyzer**
   - Connect **Build Runbook Context → Agent 1 - Incident Analyzer**
   - Set prompt type to defined/manual
   - Prompt should include:
     - title
     - description
     - reportedSeverity
     - service
     - runbookContext
   - Set a strict system message instructing the model to output only JSON with:
     - `severity`
     - `confidence`
     - `reasoning`
     - `risk_indicators`
     - `matches_known_issue`
     - `urgency_score`
   - Enable fallback model support

10. **Add primary analyzer model**
